## What This Project Is

`pseo-engine` is an open-source, plug-and-play programmatic SEO engine.
It generates hundreds to tens of thousands of keyword-targeted static landing pages
for any site type (SaaS, blog, ecom, local business, app download, directory, news, portfolio)
without ever modifying the site's design. Only content slots change. Design is untouched.
This is a hard constraint throughout the entire build.

The engine is driven by a single `keywords.md` file that is:
1. Generated by an interactive CLI wizard
2. Human-editable before the build runs
3. Consumed by the generator as the sole source of truth for what pages to create

## Build Order

Execute in this exact sequence. Run tests after each numbered step before proceeding.

### Step 1 — Complete `engine/site-types/registry.js`
The file is scaffolded. Fill in:
- `getSiteType(id)` — reads the correct JSON file, throws descriptive error with valid IDs listed if not found
- `listSiteTypes()` — returns array of {id, displayName, description} from all JSON files
- `validateTemplateAgainstSiteType(manifest, siteTypeId)` — warns (not errors) on unknown slots

### Step 2 — Complete `engine/keywords.js`
Two classes to build:

**`KeywordWizard`** — interactive CLI using `inquirer`:
- Step 1: site name, one-line description, target region
- Step 2: 3-10 seed keywords (comma-separated input)
- Step 3: dimension selection (multi-select checkbox list):
  Location, Job Role, Product Category, Price Range, Platform/Technology,
  Persona/Audience, Feature/Use Case, Seniority Level, Industry/Vertical, Custom
- Step 4: dimension values (prompted per selected dimension)
- Step 5: intent type selection (Transactional, Commercial, Informational, Navigational)
- Step 6: exclusion patterns + minimum priority threshold
- Step 7: volume estimation method (none/DataForSEO/Semrush)
After all inputs: build Cartesian product, score, cluster, write keywords.md, print summary.

**`KeywordLibraryParser`** — reads keywords.md:
- Parses the header metadata block (HTML comment block at top)
- Parses each `## [Cluster ID: ...]` section into Cluster objects
- Parses each table row into KeywordRecord objects
- Validates URL slugs (no spaces, no uppercase, no special chars, no duplicates)
- Returns a `KeywordLibrary` object
- Detects manual edits via `lastBuildHash` comparison, warns user

### Step 3 — Complete `engine/ingest.js`
`DataIngestor` class with four source types:
- `keywords-md`: reads via KeywordLibraryParser (primary/default)
- `csv`: reads CSV with configurable column mapping from config
- `json`: reads JSON array from file
- `api`: GET/POST to endpoint, caches to disk for 24h offline rebuilds
All sources normalise to same `KeywordRecord` format.
Deduplicate on urlSlug. Log skipped records with reason codes.

### Step 4 — Complete `engine/cluster.js`
`ClusterBuilder` class:
- Takes KeywordLibrary, builds directed ClusterMap graph
- Hub pages link to all spokes, spokes link back to hub + related spokes
- Cross-cluster links where clusters share a modifier value
- Detects orphan pages (zero inbound links) and isolated clusters
- Exports ClusterMap consumed by generate.js and linker.js

### Step 5 — Complete `engine/content-providers/template-only.js`
Default provider, no LLM required:
- `generatePage(slots, context, rules)` — generates all slots using rule-based string construction
- Uses modifier banks from `/data/modifiers.json`
- For `body_content`: structured HTML paragraphs with keyword-seeded variable insertion
- For `faq_block`: generates `<dl><dt>question</dt><dd>answer</dd></dl>` HTML
- For `meta_title`: `{primaryKeyword} — {siteName}` pattern, trimmed to 60 chars
- For `meta_description`: modifier-specific description, trimmed to 155 chars
- Guarantees no two pages in the same cluster have identical slot content
- No API keys needed. This is the zero-cost baseline.

### Step 6 — Complete `engine/content-providers/claude.js`
`ClaudeContentProvider`:
- Uses `@anthropic-ai/sdk`, requires `ANTHROPIC_API_KEY` env var
- `generatePage()`: single API call, structured system prompt (template in PROMPT_TEMPLATE below)
- Returns ONLY JSON with slot names as keys — no markdown, no preamble
- Validates response against contentRules, retries once on failure with error context appended
- Rate-limits to 10 req/s (configurable)
- Caches responses in `/output/.content-cache/` keyed by hash of context object
- Logs token usage per page, totals in build report

PROMPT_TEMPLATE for Claude provider:
```
You are a content writer for {{siteDescription}}.
Generate SEO content for a landing page targeting the keyword: "{{primaryKeyword}}".

Site type: {{siteType}}
Brand voice: {{brandVoice}}
Page URL: /{{urlSlug}}
Target persona: {{targetPersona}}
Search intent: {{clusterIntent}}

Return ONLY a valid JSON object with these exact keys and no other text:
{{slotsWithRules}}

Rules:
- Content must be specific to the primary keyword. No generic filler.
- Keyword density in body_content must not exceed 3%.
- FAQ questions must be real user search queries.
- meta_title must be under {{metaTitleMaxChars}} characters.
- meta_description must be under {{metaDescMaxChars}} characters.
- body_content must be valid HTML (not markdown).
- All content must sound human-written.
```

### Step 7 — Complete `engine/content.js`
`ContentEngine` class:
- Reads `contentProvider` from config
- Instantiates correct provider, runs healthCheck() before generation
- Falls back to template-only if provider fails after 2 retries
- Enforces maxConcurrent limit (default 5) to prevent rate limit errors
- Reports per-cluster stats in build report

### Step 8 — Complete `engine/generate.js`
`PageFactory` class — the engine core:
- Load template file + manifest for each cluster
- Build context object: KeywordRecord + site type profile + config values
- Call ContentEngine.generatePage() for slot content
- Inject slots using `{{slot}}` (text, HTML-escaped) and `{{{slot}}}` (HTML, unescaped) syntax
- Call SEOModule.buildHead() and replace template's existing `<head>` block
- Inject JSON-LD block from SEOModule.buildStructuredData()
- Write via correct framework adapter
- Generation order: hub pages FIRST, then spokes
- Per-page try/catch — single failure must not abort build
- Log failures to `/output/build-errors.log`

### Step 9 — Complete `engine/seo.js`
`SEOModule`:
- `buildHead(context, manifest)` — complete `<head>` block with title, meta description,
  canonical, OG tags, Twitter card, hreflang (if multi-locale), robots meta
- `buildStructuredData(context, manifest)` — JSON-LD by schemaType:
  Service, Article, Product, FAQPage, LocalBusiness, SoftwareApplication, BreadcrumbList
- FAQPage: parse `<dt>` as questions, `<dd>` as answers from faq_block slot
- BreadcrumbList: always inject alongside primary schema, constructed from URL hierarchy
- Robots meta: index/follow for qualifying pages, noindex/nofollow below content threshold

Schema templates are in `engine/seo.js` (already scaffolded). Fill them in completely.

### Step 10 — Complete `engine/linker.js`
`InternalLinker`:
- Hub → all spokes (anchor = spoke h1)
- Spoke → hub (anchor = cluster name) + 3-5 related spokes by token similarity
- Cross-cluster links where clusters share a modifier (up to maxLinksPerPage)
- Output `{{internal_links}}` HTML block per page:
  ```html
  <nav class="pseo-internal-links" aria-label="Related pages">
    <ul><li><a href="/slug">Anchor Text</a></li></ul>
  </nav>
  ```
- Never link to noindexed pages
- Output LinkAudit.json: totals, orphan list, cross-cluster count

### Step 11 — Complete `engine/validate.js`
Run after generation. Check table:

| Check                          | Level | Action         |
|-------------------------------|-------|----------------|
| Word count < minUniqueWordCount | FAIL  | noindex + log  |
| Jaccard similarity > 0.6       | WARN  | log only       |
| Keyword density > 3%           | WARN  | log only       |
| Required slot empty            | ERROR | skip page      |
| URL collision                  | ERROR | skip second    |
| meta_title > 60 chars          | WARN  | log only       |
| meta_description > 155 chars   | WARN  | log only       |
| Invalid JSON-LD                | WARN  | log only       |
| Zero inbound internal links    | WARN  | log (orphan)   |
| HTML validation failure        | WARN  | log only       |

Output ValidationReport.json to /output/.

### Step 12 — Complete `engine/sitemap.js`
- One sitemap per cluster when segmentByCluster: true
- Sitemap index at /output/sitemap-index.xml
- Respect 50,000 URL / 50MB limits
- Priority: hub=0.9, priority-1 keywords=0.8, priority-2=0.6, priority-3=0.4
- Exclude noindexed pages
- Generate robots.txt with Sitemap directive

### Step 13 — Complete all framework adapters
`adapters/html.js` — write to /output/[slug]/index.html
`adapters/nextjs.js` — write /output/pages/[...slug].jsx with getStaticPaths + getStaticProps
`adapters/astro.js` — write /output/src/pages/[slug].astro with frontmatter
`adapters/nuxt.js` — write /output/pages/[slug].vue with definePageMeta

### Step 14 — Complete `engine/cli.js`
Wire all modules. Commands:
- `npx pseo-engine keywords` — runs KeywordWizard, writes keywords.md
- `npx pseo-engine init` — scaffolds pseo.config.json interactively
- `npx pseo-engine build [--config path] [--dry-run] [--cluster id] [--force]`
- `npx pseo-engine validate` — runs validator only on existing /output
- `npx pseo-engine report` — reads ValidationReport.json + LinkAudit.json, prints summary
- `npx pseo-engine clean [--confirm]` — deletes /output
Colour output (chalk). Progress bars (ora). Exit code 1 on ERROR-level failures.

### Step 15 — Build content providers: openai.js and local-llm.js
Mirror claude.js structure. openai.js uses gpt-4o. local-llm.js targets Ollama at localhost:11434.

### Step 16 — Complete blog and ecom examples
Fill in pseo.config.json, keywords.md, and templates for both examples.

### Step 17 — Write DOCUMENTATION.md
13 sections as specified in this file (see Documentation Spec section below).

### Step 18 — Write README.md
One-liner, why it exists, real feature list only, 3-command quickstart, links to docs and examples.

## Keywords.md Format Spec

The parser depends on this exact format. Do not deviate.

```markdown
# Keyword Library
<!-- DO NOT EDIT THIS HEADER BLOCK -->
<!-- site: Site Name -->
<!-- generated: ISO8601 timestamp -->
<!-- siteType: saas-landing -->
<!-- totalKeywords: 347 -->
<!-- totalClusters: 6 -->
<!-- lastBuildHash: sha256-of-content -->

---

## [Cluster ID: cluster-id-slug] Human Readable Cluster Name
**Hub URL:** /hub-page-slug
**Hub Page Title:** Full Hub Page Title — Site Name
**Schema:** Service
**Intent:** transactional
**Priority:** HIGH
**Page Count:** 100

| Keyword | URL Slug | Primary Modifier | Secondary Modifier | Priority | Notes |
|---------|----------|-----------------|-------------------|----------|-------|
| full keyword phrase | url-slug-here | modifier1 | modifier2 | 1 | optional note |

---
```

Parser rules:
- Header block is HTML comment format, parse with regex
- Cluster sections delimited by `## [Cluster ID: ...]` pattern
- Table rows are pipe-delimited, first row is header (skip it)
- Priority is integer 1/2/3 (1=highest)
- Notes column is optional and may be empty
- Blank lines and extra whitespace are tolerated
- Manual edits to rows are expected and must be handled gracefully

## Template Slot Syntax

Templates use Handlebars-style slot syntax:
- `{{slot_name}}` — text slot, HTML-escaped on injection
- `{{{slot_name}}}` — HTML slot, injected raw (unescaped)

Standard slots (all site types):
| Slot | Type | Max Length | Required |
|------|------|-----------|---------|
| meta_title | text | 60 chars | yes |
| meta_description | text | 155 chars | yes |
| h1 | text | 70 chars | yes |
| hero_subtext | text | 160 chars | yes |
| body_content | HTML | 600 words | yes |
| faq_block | HTML | 6 items | recommended |
| cta_text | text | 50 chars | recommended |
| breadcrumbs | HTML | auto | auto |
| internal_links | HTML | auto | auto |

Site-type-specific slots are declared in each template's `.manifest.json`.

## Template Manifest Spec

Every template file must have a sibling `.manifest.json`:

```json
{
  "templateId": "unique-id",
  "version": "1.0.0",
  "siteType": "saas-landing",
  "frameworkAdapter": "nextjs",
  "description": "What this template is for",
  "requiredSlots": ["meta_title", "meta_description", "h1", "hero_subtext", "body_content"],
  "optionalSlots": ["faq_block", "cta_text", "social_proof_stat"],
  "customSlots": [],
  "schemaType": "Service",
  "canonicalStrategy": "self",
  "contentRules": {
    "meta_title": { "minChars": 40, "maxChars": 60, "mustContain": "primaryKeyword" },
    "meta_description": { "minChars": 120, "maxChars": 155, "mustContain": "primaryKeyword" },
    "h1": { "minChars": 20, "maxChars": 70, "mustContain": "primaryKeyword" },
    "body_content": { "minWords": 250, "maxWords": 600, "maxKeywordDensity": 0.03 },
    "faq_block": { "minItems": 3, "maxItems": 6 }
  }
}
```

## Definition of Done

The build is complete only when:

1. `node engine/cli.js start` collects a page URL/file + keywords and handles all setup
2. `npx pseo-engine keywords` produces a valid keywords.md for any domain
3. `npx pseo-engine build --config examples/saas-example/pseo.config.json` generates all 158 pages with zero ERROR-level failures
4. `npx pseo-engine build --config examples/blog-example/pseo.config.json` generates all 50 pages cleanly
5. `npx pseo-engine build --config examples/ecom-example/pseo.config.json` generates all 80 pages cleanly
6. Every generated page has: valid HTML, correct JSON-LD, canonical URL, meta_title ≤60 chars, meta_description ≤155 chars, at least one inbound internal link, BreadcrumbList schema
7. `npx pseo-engine report` shows zero orphan pages across all examples
8. DOCUMENTATION.md covers all 13 sections
9. README.md complete

## Hard Constraints (repeat for emphasis)

- The engine NEVER modifies any file outside `/output` and `/data/keywords.md`
- Template design is NEVER touched. Only slot content changes.
- Every module wraps page-level operations in try/catch. One failure never kills the build.
- The `template-only` content provider must work with zero API keys.
- All generated HTML must pass basic validity (no unclosed tags, valid JSON-LD).
- Sitemaps must exclude noindexed pages.
- Internal links must never point to noindexed pages.
